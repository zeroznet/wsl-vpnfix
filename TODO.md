Persist the snapshot to `/run/wsl-vpnfix/saved-routes.json` (or equivalent) so `systemctl restart wsl-vpnfix` mid-init does not require `wsl --shutdown`. Tracked as audit finding F-014. Brainstorm before plan; format / ACL / lifecycle on `wsl --unregister` are open design decisions.

At next upgrade event capture `Get-ScheduledTaskInfo -TaskName wsl-vpnfix | Select LastRunTime,LastTaskResult,NextRunTime,NumberOfMissedRuns` + `(Get-ScheduledTask -TaskName wsl-vpnfix).State` + `wsl -l -v` to determine: (a) Start-ScheduledTask never fired the action (`LastRunTime = Never`), (b) action fired but failed (`LastTaskResult != 0`), or (c) corp GPO disabled the task (`State = Disabled`). Likely fix: replace the `Start-ScheduledTask` call on line 164 with `Start-Hidden`; the registered task stays in place for subsequent logons.
- [ ] **Known-flake: `TestTap_Idempotent` in `internal/netlink/tap_test.go`.** First seen 2026-05-11 in CI run `25657040859` (post-merge of PR #22, identical code passed PR-run `25657015258` minutes earlier). Failure mode: second `CreateTap(name, mac)` call sees kernel-default random MAC instead of the one set by the first call's `LinkSetHardwareAddr`. Race between netlink ACK of `LinkSetHardwareAddr` and follow-on `LinkByName` reading the IFLA_ADDRESS update from the in-kernel cache, surfaces under CI runner load. Local repro: 0/10 in dev container. Production impact: zero — `CreateTap` is called exactly once during distro init, so the second-call path never fires outside tests. Fix when this recurs (~20-PR window): add a 3-iteration verify-loop in `CreateTap` after `LinkSetHardwareAddr` (re-read MAC, sleep 20ms, retry) and require `time` import; alternatively make the test wait/retry instead of touching production code.
