## Bug fixes that came out of this smoke run

The first import attempt hit four real defects that this smoke session uncovered. All are committed on `main`. Each was reproducible on the home PC and would have surfaced identically on the work PC.

| Commit | Fix |
|---|---|
| `974e9b4` (`stage`) | gvproxy.exe staged from `/etc/wsl-vpnfix/wsl-gvproxy.exe` (Linux 9P) to `/mnt/c/Users/Public/.wsl-vpnfix/wsl-gvproxy.exe` (Windows NTFS via DrvFs). Win 11 25H2 demand-paged code over 9P fails with `EXCEPTION_IN_PAGE_ERROR (0xc0000006)` and crashes the .exe before the first stdin read. NTFS pages-in works reliably. |
| `2b54529` (`spawn`) | `-ssh-port=-1` passed to gvproxy via the gvforwarder `stdio:` URL. Without it, gvproxy v0.8.8's default SSH listener at `127.0.0.1:2222` fights any pre-existing process on that port and exits with `cannot add network services`. We do not use SSH forwarding at all. |
| `e30d045` (`spawn`) | Workaround for upstream gvisor-tap-vsock v0.8.8 regression: `cmd/gvproxy/config.go:125` declares `-listen-stdio` but never wires it into `config.Interfaces.Stdio`, so `cmd/gvproxy/main.go:291`'s stdio acceptor goroutine never starts (silent TX-only bridge). Workaround: stage a YAML `gvproxy.yaml` in the same Windows directory and pass `-config=C:\Users\Public\.wsl-vpnfix\gvproxy.yaml` — the YAML sets `interfaces.stdio` directly, bypassing the broken CLI wiring. |
| `0893652` (`netfilter`) | Drop the `ip saddr 192.168.127.0/24` qualifier from the wsltap MASQUERADE rule. Sibling distros (Ubuntu, etc.) have eth0 IPs in `172.29.x.x`; with the qualifier scope, MASQUERADE skipped their packets, gvproxy saw the original source IP, and could not route the reply (its user-mode stack only knows `192.168.127.0/24`). Phase A's F-007 hypothetical "origin masking" finding was the wrong fix for the actual functional requirement. |

## Master-spec implications

`docs/superpowers/specs/2026-05-08-wsl-vpnfix-design.md` section 3.5 currently includes `F-007` as an example audit finding stating that the MASQUERADE rule should be `oifname "wsltap" ip saddr 192.168.127.0/24`. That example is now incorrect — the actual MASQUERADE in production is `oifname "wsltap"` only, and any audit doc must reflect that. Phase C SECURITY-AUDIT.md should record this as a documented design decision (with the `0893652` commit + this smoke note as the evidence trail), not as a standing finding.
